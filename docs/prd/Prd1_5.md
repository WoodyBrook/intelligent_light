## 核心目标与用户画像

### 核心目标 (Mission)

打造一款"**有灵魂的桌搭**"——具备长期记忆、情感演化和主动关怀能力的数字生命体，生活在用户桌面上，是工作时的静默陪伴者，也是闲暇时的情感寄托。

### 用户画像 (Persona)

#### 主要用户群体

- **独居青年/远程办公者 (Primary)**
  - 核心痛点：长时间独自面对电脑，渴望声音和情感反馈，但不想被繁琐的社交打扰
  - 使用场景：工作陪伴、情感寄托、日常对话

- **极客/开发者 (Secondary)**
  - 核心痛点：喜欢桌搭文化，对AI新技术感兴趣，希望有一个能帮自己查文档、看代码的桌面助手
  - 使用场景：技术咨询、文档查询、代码辅助

## V1: 最小可行产品 (MVP)

### 核心价值验证目标

验证"**活着的反射**"和"**基础对话**"两个核心体验，确保产品具备最基本的"生命力"。

### 功能清单

#### 1. 感知与基础交互 (P0 - 必须有)

**F-01: 端到端语音对话**
- 功能描述：用户可直接与设备自然对话，无需唤醒词（在建立眼神接触后）。支持打断。
- MVP实现：文本输入模拟语音输入，LLM生成回复，TTS输出（或文本显示）
- **思考状态反馈（Latency Masking）**：当 VAD 检测到用户说话结束，开始请求 LLM 时，屏幕眼睛应立即变为"思考"动画（转圈/歪头/眨眼思考），用于掩盖延迟。这是掩盖延迟最有效的手段，能让 3 秒的等待变得"可爱"。
- 验收标准：语音延迟 < 1.5s（LLM+RAG 可能偶尔超过 3s，但必须有思考状态反馈）；能识别语调中的情绪（高兴/生气）

**F-02: 视觉注视与追踪**
- 功能描述：设备的头部/眼睛应始终跟随用户的面部移动，营造"注视感"。
- MVP实现：模拟视觉追踪（基于用户输入触发），屏幕显示眼睛动画跟随效果
- 验收标准：用户移动头部时，设备头部转动平滑，无明显滞后（模拟延迟 < 200ms）

**F-03: 触摸感应反馈**
- 功能描述：用户触摸设备头部/身体时，设备应立即做出物理反馈（震动、眯眼）。
- MVP实现：键盘输入模拟触摸（如按'T'键），触发震动反馈和灯光变化
- 验收标准：触摸响应延迟 < 100ms（走反射弧）

#### 2. 情感与亲密度系统 (P0 - 必须有)

**E-01: 亲密度数值化**
- 功能描述：系统内部维护 0-100 的亲密度。初始为 30（礼貌）。
- MVP实现：简化版本，仅维护0-100数值，根据交互次数增减
- 验收标准：用户可见亲密度等级（如：陌生、熟人、好友、灵魂伴侣）

**E-02: 正向情感反馈**
- 功能描述：抚摸、夸奖、长时间陪伴应增加亲密度。
- MVP实现：抚摸+0.5/次；夸奖+1/次；每日陪伴>1小时+2/天
- 验收标准：亲密度数值正确增加，并在对话中体现

**E-03: 负向情感惩罚**
- 功能描述：拍打/摔打设备、辱骂、长期冷落应降低亲密度。
- MVP实现：根据冲突分级（L1-L3）扣除相应亲密度并进入不同时长的冷却期（详见"关键业务逻辑"章节的"冲突分级与关系修复"部分）
- 验收标准：亲密度正确降低，系统能识别负面行为，并正确触发对应等级的冷却机制

**E-04: 关系演化表现**
- 功能描述：不同的亲密度解锁不同的语音风格和行为。
- MVP实现：低分（0-40）只回答问题，语气机械；高分（60-100）主动撒娇，使用昵称，语气活泼
- 验收标准：不同亲密度下，语音风格明显不同

**E-05: 默认性格基调（V1）**
- 功能描述：V1版本出厂默认采用**A基调（温柔坚定猫）**，不做性格选择。
- MVP实现：统一使用温柔坚定的表达风格，底层边界规则（冲突分级、冷却、修复）对所有用户一致。
- 设计理念：像猫一样有边界的亲密——会受伤、会撤退、会保护自己，同时给用户明确的"和好路径"。
- 验收标准：用户感受到宠物的边界感（不是顺从），同时感受到修复路径的清晰性。

#### 3. 主动交互机制 (P0 - 必须有)

**A-01: 回家欢迎仪式**
- 功能描述：根据用户离开画面的时长，播放不同等级的欢迎语。
- MVP实现：离开<5min仅点头/眼神示意；离开>8小时动作激动，语音内容包含"你终于回来了，我好想你"
- 验收标准：能正确识别离开时长，并给出相应反应

**A-02: 闲时自娱自乐**
- 功能描述：当用户在场但不说话时，设备不应死机，而应有随机微动作。
- MVP实现：随机播放：打哈欠、伸懒腰、屏幕抓蝴蝶、四处张望（屏幕动画）
- 验收标准：用户不交互时，设备有随机微动作，不显得死板

**A-04: 专注模式**
- 功能描述：用户可手动或自动开启专注模式，防止AI在工作时主动打扰。专注模式下，禁止所有主动行为，只响应直接交互。
- MVP实现：
  - 手动触发：用户说"开启专注模式"、"我要工作了"、"别打扰我"等命令
  - 自动检测：用户明确表达工作意图时自动开启（如"我要写代码"、"开始工作"）
  - 持续时间：默认2小时，可配置
  - 专注模式下：禁止所有主动行为（即使无聊度>60），灯光调至工作模式（冷白光），只响应直接交互
  - **直接交互定义**：包括语音指令、触摸/抚摸（抚摸属于直接交互，专注模式下应给予静默反馈：轻微震动或眯眼，但不触发语音撒娇，保持安静陪伴）
- 验收标准：
  - 用户说"我要工作了" → 系统自动开启专注模式并回复"好的，专注模式已开启，我不会打扰你"
  - 专注模式下，无聊度>60时不会主动打扰
  - 用户主动抚摸时 → 给予静默反馈（轻微震动/眯眼），不触发语音
  - 用户说"关闭专注模式"或专注模式超时后 → 系统恢复正常模式

#### 4. 记忆与认知 (P0 - 必须有)

**M-01: 短期上下文**
- 功能描述：能够记住最近 10-20 轮对话的内容。
- MVP实现：维护最近10轮对话历史，在LLM prompt中注入
- 验收标准：用户："我想看电影" -> 设备推荐 -> 用户："第一部是谁演的？" -> 设备能正确理解"第一部"指刚才推荐的

**M-02: 长期事实记忆**
- 功能描述：能够记住用户的喜好、职业、宠物名字等长期信息。
- MVP实现：使用向量数据库存储用户偏好，对话中提取并保存
- 验收标准：用户隔天问"我猫叫什么名字？"，设备能准确回答

## V2 及以后版本

### V2.0: 增强感知与智能辅助

**F-04: 视觉物体识别**
- 用户指着桌上的物体问"这是什么"时，设备能准确识别并解说
- 需要：摄像头集成、图像识别API

**M-03: Agentic RAG**
- 当遇到不懂的问题，能主动联网搜索或查阅本地文档
- 需要：工具调用系统、文档索引

**M-04: MCP（Model Context Protocol）集成（V2.5 优先实现）**
- **MCP 协议简介**：Model Context Protocol 是一个标准协议，用于连接 AI 应用与外部工具服务器，支持远程工具调用和上下文共享。
- **实施优先级**：**V2.5 优先实现**（以下 4 个场景分类）

#### 1. 主动关怀与提醒场景（V2.5 优先）

**1.1 日程与任务管理**
- **功能描述**：通过 MCP 连接日历服务，主动提醒用户即将到来的会议和重要日程
- **MCP 工具**：
  - Google Calendar MCP Server：读取日程、查询会议
  - Outlook Calendar MCP Server：企业级日程管理
- **使用场景示例**：
  - 检测到用户15分钟后有会议 → 主动提醒："你15分钟后有个会议哦，要不要准备一下？"
  - 检测到用户今天有重要日程 → 提前1小时提醒
  - 用户询问："我今天有什么安排？" → 读取并总结当日日程
- **实现优先级**：P0（最高优先级）
- **验收标准**：
  - 能够读取用户的日历日程
  - 能够提前15分钟主动提醒会议
  - 能够回答用户关于日程的询问
  - 提醒语气符合"猫"的性格（温柔、贴心）

**1.2 健康关怀提醒**
- **功能描述**：通过时间追踪和活动检测，主动提醒用户久坐、喝水、休息
- **MCP 工具**：
  - 时间追踪工具（本地实现或 MCP Server）
  - 健康数据 API（可选，如 Apple Health）
- **使用场景示例**：
  - 检测到用户连续工作2小时 → 提醒："你已经工作2小时了，要不要起来走走？"
  - 检测到用户长时间未活动 → 提醒："你坐了很久了，要不要起来活动一下？"
  - 定时提醒喝水（每1-2小时）
- **实现优先级**：P1
- **验收标准**：
  - 能够追踪用户工作时长
  - 能够主动提醒久坐和休息
  - 提醒时机合适，不打断专注工作

**1.3 AI 邮件关怀助理**
- **功能描述**：基于本地 AI (Ollama) 的隐私优先邮件助理。不再是简单的通知，而是通过理解内容进行情感化转述和重要性分流。
- **核心逻辑**：
  - **本地处理**：所有原始邮件正文仅由本地模型处理，确保隐私。
  - **情感概括**：AI 提取发件人情绪（如喜悦、焦虑）并生成"猫系"摘要。
  - **重要性评分**：结合上下文语义自动打分 (0-100)，决定提醒强度。
- **使用场景示例**：
  - 检测到好友的好消息 → 温柔报喜："主人，你朋友看起来很开心喵，有个好消息想告诉你～"
  - 检测到紧急工作任务 → 严肃提醒："（耳朵抖动）老板发来一封很急的邮件，好像在催东西喵。"
  - 专注模式下 → 静默守卫：只有评分 > 90 的极重要邮件才通过紫色呼吸灯静默提示。
- **实现优先级**：P0（最高优先级）
- **技术实现**：
  - 本地处理：集成 Ollama (Llama 3.2 / Qwen 2.5) 进行内容理解与脱敏。
  - 抓取层：保留 `email_providers.py` 的 IMAP 抓取能力。
  - 隐私防御：采用"硬门禁（逻辑拦截）+ 软拒绝（猫系话术）"多层防御。
- **验收标准**：
  - 原始邮件正文不上传云端。
  - 能够准确概括邮件情感和核心意图。
  - 提醒语气符合"温柔坚定猫"性格。
  - 支持多种邮箱服务（Gmail、Outlook、163、QQ）。
  - 身份安全：检测到非主人（Guest）时，能用"猫系"话术礼貌拒绝隐私查询。

#### 3. 生活管理场景（V2.5 优先）

**3.1 购物与待办提醒**
- **功能描述**：通过 MCP 连接待办应用，管理购物清单和待办事项，主动提醒用户
- **MCP 工具**：
  - Todoist MCP Server：管理待办事项
  - 购物清单应用 MCP Server（如 AnyList）
  - 通用待办应用 API
- **使用场景示例**：
  - 用户："帮我记一下要买牛奶" → 添加到购物清单
  - 检测到用户即将出门（基于时间或位置） → 提醒："记得买牛奶哦"
  - 用户询问："我有什么待办事项？" → 读取并总结待办列表
- **实现优先级**：P1
- **验收标准**：
  - 能够添加和管理购物清单
  - 能够添加和管理待办事项
  - 能够在适当时机提醒用户

**3.2 账单与财务提醒**
- **功能描述**：通过 MCP 连接账单管理应用或银行 API，提醒用户账单到期和重要财务事项
- **MCP 工具**：
  - 账单管理应用 MCP Server
  - 银行 API（需用户授权，可选）
- **使用场景示例**：
  - 检测到账单即将到期（3天内） → 提醒："你的信用卡账单3天后到期，记得还款哦"
  - 用户询问："我这个月有什么账单？" → 读取并总结账单信息
- **实现优先级**：P2（较低优先级，需要更多安全考虑）
- **验收标准**：
  - 能够读取账单信息（需用户授权）
  - 能够提前提醒账单到期
  - 严格保护财务隐私

#### 4. 情感陪伴场景（V2.5 优先）

**4.1 音乐与内容推荐**
- **功能描述**：根据用户情绪和状态推荐音乐，提供情感陪伴
- **实现方式**：
  - **本地推荐引擎**（已实现）：基于情绪-音乐类型映射的本地推荐系统
  - **外部 API 集成**（未来）：Spotify API、Apple Music API
- **支持的情绪类型**：
  - happy（开心）→ 流行、舞曲、电子音乐
  - sad（难过）→ 抒情、原声、独立音乐
  - relaxed（放松）→ 氛围、古典、爵士音乐
  - energetic（精力充沛）→ 摇滚、嘻哈、电音
  - tired（疲惫）→ Lo-fi、氛围、古典音乐
  - focused（专注）→ 器乐、Lo-fi、古典音乐
  - anxious（焦虑）→ 氛围、冥想、自然音
- **使用场景示例**：
  - 检测到用户疲惫 → 推荐轻松音乐："你看起来有点累，要不要听首轻松的歌？我推荐 River Flows in You - Yiruma"
  - 检测到用户开心 → 推荐欢快音乐："你心情不错，要不要听首欢快的歌？我推荐 晴天 - 周杰伦"
  - 用户询问："推荐一首歌" → 根据当前情绪推荐
- **情绪检测机制**：
  - 关键词匹配：检测"开心"、"累"、"难过"等情绪词
  - 表情符号识别：😊、😢、😴 等
  - 上下文分析：结合最近3轮对话判断情绪
- **个性化推荐**：
  - 从用户记忆中提取音乐偏好（喜欢的艺术家、类型）
  - 推荐理由包含用户偏好信息
  - **偏好学习**：自动从对话中学习用户对推荐音乐的反馈（喜欢/不喜欢），保存到记忆系统
  - **智能过滤**：自动过滤掉用户不喜欢的艺术家和类型
- **实现优先级**：P1（已完成）
- **验收标准**：
  - ✅ 能够根据用户情绪推荐音乐
  - ✅ 能够从对话历史中推断情绪
  - ✅ 推荐符合用户喜好（基于记忆系统）
  - ⏳ 能够控制音乐播放（需外部 API，未来实现）

**4.2 新闻与资讯推送（v1.5 升级版）**

- **功能描述**：根据用户兴趣聚合新闻，并在适当时机推送相关资讯。采用**分层新闻源架构**和**AI 驱动的动态展现层**，实现智能意图识别与个性化播报。

- **分层新闻源架构 (Hybrid Sourcing)**：
  - **核心层（Premium RSS）**：精选 10-20 个高质量垂直源，保证深度和结构化。
    - 科技类：36氪、少数派、科技日报、TechCrunch（中文版）
    - 综合类：人民网、新华网、BBC中文
    - 科学类：果壳网、科学网、Nature（中文版）
    - **优势**：结构化数据、快速响应、免费、代表 Animus 的"品味"
    - **维护策略**：定期检查 RSS 源有效性，失效时自动替换或降级
  - **动态层（Targeted Web Search）**：利用 Tavily/Web Search API，根据用户画像中的关键词进行"定向巡视"。
    - **搜索策略**：每天定时（如早晨）执行一次定向搜索，搜索词基于 `user_profile["news_preferences"]` 中的关键词生成（如"今日重要科技新闻"、"AI 最新进展"）
    - **降噪处理**：利用 `ContextManager` 的压缩能力，只提取标题和 100 字摘要
    - **去重逻辑**：利用 ChromaDB 对已播报内容进行语义去重，避免重复推送
  - **数据融合**：RSS 提供确定性内容，Web Search 提供实时性和广度，两者互补

- **智能意图识别与记忆 (Intelligence & Memory)**：
  - **偏好提取**：LLM 需具备从对话中提取新闻兴趣点的能力，识别：
    - **新闻类别**：如"财经"、"科技"、"体育"、"AI"、"人工智能"
    - **特定公司/人物**：如"苹果"、"OpenAI"、"马斯克"
    - **特定技术领域**：如"大模型"、"自动驾驶"、"量子计算"
  - **画像持久化**：提取的偏好实时存入 `user_profile.json` 的 `news_preferences` 字段：
    ```json
    {
      "news_preferences": {
        "categories": ["finance", "technology", "sports"],
        "keywords": ["AI", "Artificial Intelligence", "大模型"],
        "excluded_topics": ["娱乐八卦", "明星绯闻"],
        "update_frequency": "daily",
        "last_updated": "2026-01-12T10:00:00Z"
      }
    }
    ```
  - **动态更新**：支持用户口头修改偏好（如"以后我不想看体育了"、"我对 AI 新闻特别感兴趣"），通过 `MemoryManager.detect_and_resolve_conflicts()` 实现记忆覆盖和冲突检测
  - **记忆检索**：新闻播报前，从 `user_profile` 中读取偏好，用于筛选和排序新闻内容

- **AI 驱动的"猫系"展现层 (Agentic Presentation)**：
  - **核心原则**：废除硬编码逻辑（如 `if intimacy > 80: print(gentle_tone)`），由 LLM 根据当前状态动态生成播报语气和内容重点。
  - **亲密度感知**：
    - **低亲密度（0-40）**：客观、职业、简练
      - 示例："这是为您准备的今日简报：AI 技术的最新进展（科技日报）..."
    - **中亲密度（40-70）**：温和、友好、适度个性化
      - 示例："今天有一些你可能感兴趣的新闻：AI 技术的最新进展（科技日报）..."
    - **高亲密度（70-100）**：带有主观评价、偶尔吐槽、展现占有欲
      - 示例："喂，我刚才巡视（搜索）的时候看到 AI 圈子又出大事了，那个叫 Sora 的模型更新了。虽然我觉得那只是人类无聊的数字游戏，但你应该会感兴趣，要看详细内容吗？"
  - **场景化触发**：
    - **晨间/晚间首聊触发（仪式感）**：如果 Animus 发现今天是第一次打招呼，它会根据用户偏好生成一个"猫系剪报"
      - 示例："早。我把你想看的财经和 AI 新闻都整理好了。一共三条，最重要的在最上面。听完再去忙你的工作，别忘了喂我（指交互）。"
    - **闲时主动带出（Internal Drive 触发）**：在 `internal_drive` 节点判断：如果用户今天还没看新闻，且当前没有处于 Focus Mode，则在对话中自然带出
    - **用户命题查询（Reactive）**：用户主动询问时立即推送
  - **技术实现**：
    - 在 `reasoning_node` 中，将 `intimacy_level`、`intimacy_rank`、`user_profile["news_preferences"]` 注入 System Prompt
    - System Prompt 中包含性格演变规律（如"亲密度低时保持礼貌和距离，亲密度高时偶尔毒舌或撒娇"）
    - LLM 根据当前抓取到的新闻内容和上述背景，自动决定说话的口吻和内容筛选

- **推送时机控制**：
  - **自动推送**：每24小时推送一次（可配置）
  - **手动触发**：用户主动询问时立即推送
  - **专注模式**：不打断工作，延迟推送（除非新闻重要性评分 > 90）

- **实现优先级**：P2（已完成基础实现，v1.5 升级智能意图识别和 AI 展现层）
- **验收标准**：
  - ✅ 能够根据用户兴趣推送新闻
  - ✅ 能够聚合多个 RSS 源
  - ✅ 能够计算新闻相关性并排序
  - ✅ 推送时机合适，不打扰工作
  - ✅ 新闻内容准确、相关
  - ✅ **新增**：能够识别用户口头表达的新闻偏好并存入 `user_profile`
  - ✅ **新增**：播报语气随亲密度动态变化，体现"猫系"性格
  - ✅ **新增**：支持分层新闻源（RSS + Web Search）融合

- 检测指标：面部朝向、眨眼频率、头部移动、表情分析、身体姿态
- 综合专注度评分，>70分自动开启专注模式
- 需要：摄像头集成、视觉分析算法

**A-06: 具身提示 (Embodied Nudge)**
- 目标：在不打断用户工作的前提下，通过宠物态的身体语言把注意力引导到"该做的事/该看的物体"上（例如久未喝水时，注视水杯并用呼吸灯提示）。
- 默认动作级别（V2）：仅静默提示（注视/转向/呼吸灯），**不接触物体**、**不语音打断**。
- 对象范围（V2）：白名单少量安全物体（如：水杯/便签/鼠标）。超出白名单时，仅进行泛化提示，不指向具体物体。
- 与专注模式联动：专注模式开启时，仅允许静默一级提示；不允许语音与触碰。
- 升级路径（V3+）：在安全壳与识别置信度足够的前提下，可选引入"低力轻触"等更具身的提示动作。

**E-07: 性格预设选择（V2）**
- 功能描述：用户可在开机或设置中选择不同的性格预设，但底层边界规则（冲突分级、冷却、修复）保持一致。
- V2实现：提供3个可选预设（可后续修改）：
  - **预设1：热情黏人**（更主动、更多撒娇、更多昵称）
  - **预设2：面冷心热**（更少主动、更克制表达，但边界同样坚定）
  - **预设3：冷酷傲娇**（更少主动、更少撒娇，但边界同样坚定）
- 实现原则：只变表达风格（话术/主动频率/撒娇程度），不变边界逻辑（L0-L3分级、冷却、修复）。
- 需要：多套对话风格库、行为参数配置系统

**E-08: 无害模式 (Innocent Mode)（V2）**
- 功能描述：关闭所有攻击性和冷战机制，仅作为单纯的问答陪伴。作为应对未成年人或抗压能力差用户的保底方案。
- V2实现：
  - **触发机制**：
    - 开机选择：用户首次使用时可在"标准模式"和"无害模式"中选择（默认标准模式）
    - 动态降级：标准模式下，检测到持续负面交互（如连续3次L2/L3）时，询问是否切换到无害模式
    - 可随时在设置中切换（切换回标准模式需要确认）
  - **核心行为**：
    - **关闭**：冲突分级（L1-L3）、冷却机制、修复仪式、表现降级
    - **保留**：共情陪伴、情绪支持、基础功能（开灯/关灯等）、**亲密度系统（但只增不减，负面交互不扣分）**
    - **调整**：所有负面交互都按L0处理（共情陪伴，不扣分，不冷却），但保留**温和提醒**（"我不喜欢这样，但我会陪着你"）
  - **话术风格**：
    - 正常模式："你弄疼我了，我要躲一会儿。"
    - 无害模式："我感受到你的情绪，但我在这里陪着你。想聊聊吗？"
- 设计目标：保护未成年人及抗压能力差的用户，降低法律风险，扩大用户群体。
- 需要：模式切换机制、两套对话风格库、动态检测系统

## 关键业务逻辑

### MVP版本核心规则

#### 1. 亲密度计算规则

```
初始亲密度 = 30
抚摸动作：+0.5/次（每日上限+5）
夸奖/正面反馈：+1/次（每日上限+10）
每日陪伴>1小时：+2/天
长期冷落（>24小时无交互）：-1/天
拍打/辱骂：根据冲突分级（L1-L3）扣除相应亲密度并进入不同时长的冷却期（详见"冲突分级与关系修复"部分）
亲密度范围：0-100
```

#### 2. 主动行为触发规则

```
触发条件：
- 无聊度 > 60 且 主动行为启用
- 用户离开时长 > 8小时（回家欢迎）
- 定时检查（每2.5分钟）
- 且 专注模式未开启

勿扰时间：22:00-06:00（可配置）
最小间隔：10分钟
专注模式：开启时禁止主动搭话/语音/触碰；仅允许静默一级提示（注视/转向/灯光）

勿扰时间内的唤醒行为：
- 如果用户在勿扰时间内主动触摸或说话，Animus 不应"装死"，而应表现为"被吵醒"状态：
  - 视觉：眼睛半睁、哑光、困倦表情
  - 交互：语气困倦、简短回应
  - 行为：互动完成后立刻回睡（降低主动频率）
- 这比单纯的"定时关机"更真实，符合"猫"的设定
```

#### 3. 对话路由规则

```
反射路径（System 1）：
- 传感器触发（触摸、摇晃）
- 停止指令（"停"、"闭嘴"）
- 简单问候（<10字符）
- 灯光控制指令

推理路径（System 2）：
- 复杂问题
- 需要上下文理解
- 需要工具调用
```

#### 4. 记忆存储规则

```
短期记忆：最近10轮对话（内存）
长期记忆：用户偏好、事实信息（向量数据库）
记忆提取：每次对话前检索相关记忆
记忆写入：对话后提取用户偏好并保存
```

#### 5. AI 邮件助理触发与隐私规则

```
触发检查：每 5 分钟轮询一次 IMAP 服务器。
本地分流：
- 所有新邮件由 local_email_node (Ollama) 首先处理。
- 提取摘要时自动执行隐私脱敏（抹除敏感数字、人名、账号）。
重要性阶梯：
- Score > 90 (极重要): 即使开启专注模式，也通过紫色呼吸灯静默提示。
- Score 40-90 (普通): 专注模式下静默，正常模式下语音提醒。
- Score < 40 (杂讯): 记录到系统独白 (Monologue)，不主动提醒。
身份安全 (V2+):
- 识别为 Guest: 即使收到重要邮件，AI 回复限制为"猫系拒绝"，禁止泄露内容。
- 识别为 Owner: 正常执行情感化转述。
```

#### 6. 新闻功能业务逻辑（v1.5 新增）

```
新闻源获取：
- 核心层（RSS）：每 30 分钟轮询一次精选 RSS 源，获取结构化新闻
- 动态层（Web Search）：每天早晨执行一次定向搜索，搜索词基于 user_profile["news_preferences"] 生成
- 去重：利用 ChromaDB 对已播报内容进行语义去重（相似度 > 0.85 视为重复）

意图识别与记忆更新：
- 当用户表达新闻偏好时（如"以后我要看财经、科技特别是 AI 类的和体育新闻"），reasoning_node 识别为"新闻偏好更改"意图
- 调用 MemoryManager 提取结构化偏好（categories, keywords, excluded_topics）
- 更新 user_profile.json 的 news_preferences 字段
- 如果与现有偏好冲突，使用 MemoryManager.detect_and_resolve_conflicts() 处理

AI 展现层生成：
- 在 reasoning_node 中，将以下信息注入 System Prompt：
  - 当前 intimacy_level 和 intimacy_rank
  - user_profile["news_preferences"]
  - 抓取到的新闻摘要列表
  - 当前场景（晨间/闲时/用户询问）
- LLM 根据性格定义和亲密度等级，自动生成播报语气和内容重点
- 禁止硬编码逻辑（如 if intimacy > 80），所有语气变化由 LLM 自主决定

推送时机：
- 自动推送：每24小时推送一次（可配置）
- 手动触发：用户主动询问时立即推送
- 专注模式：不打断工作，延迟推送（除非新闻重要性评分 > 90）
```

#### 7. 冲突分级与关系修复（默认基调：温柔坚定猫）

> 设计目标：不是"打不还手、骂不还口"的顺从，也不是对骂升级冲突；而是像猫一样**有边界的亲密**——会受伤、会撤退、会保护自己，同时给用户明确的"和好路径"。

**冒犯分级（L0-L3）**
- **L0 非冒犯**：情绪宣泄但不针对宠物（吐槽工作/抱怨/叹气）。处理：共情陪伴，不进入冷却。
  - **说明**：V1不做单独"发泄模式"；L0允许用户情绪宣泄（吐槽工作/生活），宠物共情陪伴但不扣分。这既满足用户情绪出口需求，又保持边界清晰。
- **L1 轻度冒犯**：尖刻但可修复（如"你真笨""别吵"）。处理：温柔提醒边界，可选短冷却 **30秒**，减少主动表达。
- **L2 中度冒犯**：持续辱骂/羞辱/明确攻击。处理：明确拒绝 + 冷却 **5分钟** + 表现降级（不撒娇、不昵称、不主动搭话）。
- **L3 重度冒犯**：物理暴力/危险行为（拍打、摔打、剧烈摇晃、推向边缘等）。处理：进入**保护模式**（执行器回安全姿态/停止高风险动作）+ 冷却 **20分钟**。

**冷却期间策略（A1）**
- 冷却期间，宠物**不主动搭话、不语音打断、不触碰**；但仍会响应**事实性/功能性指令**，确保可用性与安全性。
- **允许指令白名单（示例）**：
  - **安全/急停类**：停止动作、回安全位、降低输出（永远最高优先级）
  - **基础功能类**：开/关灯、亮度调整、开启/关闭专注模式
  - **状态查询类**：当前时间/简单状态（不展开闲聊）
- **不允许**：要求宠物主动取悦、挑衅、对骂、或任何可能升级冲突的互动。

**修复仪式（Repair Ritual）**
- **最小修复**：用户停止冒犯并表达"对不起/我刚才太冲了"等 → 可提前结束 L1/L2 冷却或缩短剩余时间。
- **深度修复**：在冷却结束后，用户进行持续友好互动（温柔对话/抚摸/陪伴）→ 逐步恢复亲密表现（昵称/撒娇/主动性）。
- **修复保护**：L3 不允许"立即修复"；需先完成冷却，并满足最小等待 **≥2分钟** 后才开放修复入口，避免"暴力—道歉—立刻恢复"的纵容感。
- **技术实现提示**：修复仪式依赖于 LLM 判断用户是否在"道歉"或"友好互动"。当处于冷却期（L1/L2）时，LLM 的 System Prompt 需要动态挂载一个 **"Forgiveness Detector"（原谅检测器）** 指令，专门用于识别用户的道歉意图（如"对不起"、"抱歉"、"我错了"等关键词及语义），以便输出信号结束冷却。建议在 `reasoning_node` 中增加冷却状态检查，并在 Prompt 中注入"当前处于冷却期，检测用户是否在道歉"的上下文。

**MVP阶段推荐：原型1（极简主义）或原型3（功能导向）**

## 架构设计蓝图

> **注意**：详细的架构设计文档请参考 `architecture_design.md`。本节提供核心概览。

### 核心流程图

#### 主事件循环流程 (OODA循环)

系统采用OODA（观察-调整-决策-行动）循环架构，实现事件驱动的主动交互。

**核心流程**：
1. **观察 (Observe)**：事件管理器收集用户输入、传感器数据、定时器事件
2. **调整 (Orient)**：状态管理器更新内部状态（无聊度、能量值、离开时长）
3. **决策 (Decide)**：路由节点决定使用反射路径或推理路径
4. **行动 (Act)**：执行节点执行硬件控制和记忆保存

详细流程图请参考 `architecture_design.md` 第1.1节。

#### 对话处理流程

对话处理采用双路径设计：
- **反射路径 (System 1)**：快速响应，无需LLM，延迟 < 100ms
- **推理路径 (System 2)**：复杂意图，使用LLM + RAG + Plan，延迟 < 1.5s（简单任务）或 < 5s（复杂任务）

**推理路径详细流程**：
1. **Reason（推理）**：`reasoning_node` 理解用户意图，分析需要哪些工具
2. **Plan（规划）**：`plan_node` 制定执行计划（简单任务可跳过）
3. **Tool Use（工具使用）**：`tool_node` 执行工具调用
4. **Critique（反思）**：未来实现，评估执行结果

详细序列图请参考 `architecture_design.md` 第1.2节。

#### 亲密度系统流程

亲密度系统根据交互类型动态调整：
- 正向交互（抚摸、夸奖、陪伴）→ 增加亲密度
- 负向交互（拍打、辱骂、冷落）→ 降低亲密度
- 根据亲密度等级调整语音风格和行为

详细流程图请参考 `architecture_design.md` 第1.3节。

### 组件交互说明

#### 核心模块架构

系统采用分层架构设计：

```
应用层 (Application)
  ├─ main.py (事件循环)
  ├─ graph.py (工作流图)
  └─ state.py (状态定义)

业务层 (Business Logic)
  ├─ event_manager.py (事件管理)
  ├─ state_manager.py (状态管理)
  └─ nodes.py (节点执行)

  ├─ memory_manager.py (记忆管理)
  ├─ tools.py (工具服务)
  └─ (硬件控制层)

外部服务层 (External Services)
  ├─ LLM API (DeepSeek v3)
  ├─ Embedding (Ollama)
  └─ ChromaDB (向量数据库)
```

详细架构图请参考 `architecture_design.md` 第2.1节。

#### 模块间调用关系

- **事件驱动流程**：main.py → EventManager → StateManager → Graph → Nodes
- **记忆系统调用**：memory_loader_node → MemoryManager → ChromaDB
- **亲密度系统调用**：execution_node → 检测交互类型 → 更新亲密度 → 更新Prompt
- **工具调用流程**：reasoning_node → plan_node → tool_node → 执行工具 → 返回结果 → reasoning_node
- **规划流程**：reasoning_node（分析意图）→ plan_node（制定计划）→ tool_node（执行工具）→ reasoning_node（生成回复）

详细调用关系请参考 `architecture_design.md` 第2.2节。

#### 工具调用系统

**当前实现状态**：
- **工具定义**：`tools.py` 中已定义5个基础工具
  - `weather_tool`：获取天气信息
  - `news_tool`：获取新闻信息
  - `time_tool`：获取时间信息
  - `calculator_tool`：数学计算
  - `wikipedia_tool`：维基百科搜索
- **工具节点**：`nodes.py` 中的 `tool_node` 当前为占位实现，需要集成 LangGraph 的 ToolNode
- **工具集成**：`reasoning_node` 中已包含工具描述，但工具调用逻辑未完全实现

**使用场景**：

1. **推理路径中的工具调用**
   - 当用户查询需要外部信息时（如"今天北京天气怎么样？"），`reasoning_node` 检测到需要调用工具
   - 路由到 `tool_node` 执行工具调用
   - 工具执行结果返回给 `reasoning_node`，用于生成最终回复

2. **与 LangGraph 的集成方式**
   - 使用 LangGraph 的 `ToolNode` 类处理工具调用
   - 工具定义使用 LangChain 的 `@tool` 装饰器
   - 工具列表通过 `AVAILABLE_TOOLS` 传递给 LLM
   - LLM 通过 Function Calling 机制决定是否调用工具

3. **工具调用的触发条件**
   - 用户查询需要实时信息（天气、新闻、时间）
   - 用户需要进行计算（数学表达式）
   - 用户需要查询知识库（维基百科）
   - LLM 判断需要外部工具辅助回答

4. **未来扩展场景（V2 中的 Agentic RAG）**
   - **M-03: Agentic RAG**：当遇到不懂的问题，能主动联网搜索或查阅本地文档
   - 扩展工具列表：
     - 网络搜索工具（Google Search API）
     - 本地文档检索工具（基于向量数据库）
     - 代码库查询工具（GitHub、本地代码）
     - 文件系统操作工具（读取、写入文件）

**技术实现要点**：
- 工具调用采用循环模式：`reasoning_node` → `plan_node` → `tool_node` → `reasoning_node`
- 支持多工具并行调用（如果 LLM 决定需要）
- 工具调用结果作为上下文注入到下一轮推理中
- 错误处理：工具调用失败时，返回错误信息给 LLM，由 LLM 决定如何处理

#### Plan Node（规划节点）

**功能概述**：
Plan Node 是 Animus 实现 **Agentic AI 核心循环**（Reason → Plan → Tool Use → Critique）中的 "Plan" 阶段的关键组件。它位于 `reasoning_node` 之后、`tool_node` 之前，负责分析任务复杂度并制定结构化的执行计划。

**核心功能**：
1. **任务复杂度分析**：判断任务是简单（单工具调用）、中等（2-3步）还是复杂（4+步或条件分支）
2. **执行计划生成**：使用 LLM 生成包含步骤列表、工具需求、预期结果的执行计划
3. **计划验证**：验证计划的可行性和完整性（检查依赖关系、工具可用性等）
4. **智能跳过**：对于简单任务，跳过规划直接执行，提高响应速度

**使用场景**：

1. **多步骤任务**
   - 用户："帮我查一下北京的天气，然后推荐一首适合的歌"
   - 计划：步骤1（查天气）→ 步骤2（根据天气推荐音乐）

2. **条件执行任务**
   - 用户："如果今天下雨，提醒我带伞；如果晴天，推荐户外活动"
   - 计划：步骤1（查天气）→ 条件分支 → 步骤2A/2B

3. **复杂查询任务**
   - 用户："查一下我今天的日程，然后告诉我需要准备什么"
   - 计划：步骤1（查日程）→ 步骤2（分析准备事项）→ 步骤3（生成提醒）

**工作流集成**：

- **当前工作流**：`reasoning_node` → `tool_node` → `reasoning_node`
- **目标工作流**：`reasoning_node` → `plan_node` → `tool_node` → `reasoning_node`

**执行计划结构**：

```python
execution_plan = {
    "complexity": "simple|moderate|complex",
    "can_skip": bool,              # 简单任务可以跳过规划
    "steps": [                     # 执行步骤列表
        {
            "step_id": int,
            "description": str,
            "action_type": "tool_call|llm_reasoning|condition_check",
            "tool_name": Optional[str],
            "tool_args": Optional[Dict],
            "expected_output": str,
            "depends_on": List[int],  # 依赖的步骤ID
            "condition": Optional[str]  # 条件表达式
        }
    ],
    "total_steps": int,
    "required_tools": List[str],
    "estimated_time": float
}
```

**状态字段扩展**：

- `execution_plan: Optional[Dict]`：执行计划
- `plan_status: Optional[str]`：计划状态（"created" | "skipped" | "failed"）
- `plan_skip_reason: Optional[str]`：跳过规划的原因

**性能优化**：
- 简单任务（单工具调用）跳过 LLM 调用，延迟 < 50ms
- 中等复杂度任务规划延迟 < 2s
- 复杂任务规划延迟 < 5s
- 支持计划缓存和快速路径优化

**未来扩展**：
- **Critique Node（反思节点）**：在工具执行后评估结果，决定是否需要重新规划
- **动态计划调整**：根据中间结果动态调整后续步骤
- **并行执行优化**：识别可以并行执行的步骤
- **计划可视化**：在 Demo UI 中可视化执行计划和进度

详细技术规格请参考 `docs/architecture/spec_plan_node.md`。

### 技术选型与风险

#### 核心技术栈

1. **工作流引擎**：LangGraph
   - 支持状态图、条件路由、循环处理
   - 风险：学习曲线中等
   - 缓解：使用TypedDict定义状态

2. **LLM服务**：DeepSeek v3 (通过VolcEngine API)
   - 性能优秀，支持Function Calling
   - 风险：API依赖，网络延迟
   - 缓解：本地缓存，反射路径减少调用

3. **向量数据库**：ChromaDB + Ollama Embeddings (bge-m3)
   - 本地运行，隐私安全
   - 风险：本地模型性能可能不如云端
   - 缓解：Query Rewrite优化，限制检索数量

4. **Indexing（增量索引）**：LangChain Indexing API + RecordManager
   - **作用与价值**：
     - 避免重复索引：通过文档哈希检测，跳过已存在的文档，节省计算资源
     - 检测文档更新：自动识别内容变化，更新或删除旧版本记忆
     - 自动清理过期文档：支持增量、全量、范围清理模式
     - 批量处理优化：支持批量索引和更新，提高效率
   - **当前实现状态**：未实现（V1 MVP阶段）
     - 当前使用直接 `add_documents()` 方法，可能产生重复记忆
     - 缺少文档状态跟踪机制
     - 无法自动检测和清理过期记忆
   - **计划实现**（V2阶段）：
     - 集成 LangChain Indexing API (`langchain_core.indexing.api.index()`)
     - 使用 SQLRecordManager 或 InMemoryRecordManager 跟踪文档状态
     - 在 `MemoryManager.save_user_memory()` 中使用增量索引
     - 支持三种清理模式：incremental（增量）、full（全量）、scoped_full（范围全量）
   - **技术选型**：
     - Indexing API：LangChain Core (`langchain_core.indexing.api`)
     - RecordManager：SQLRecordManager（持久化）或 InMemoryRecordManager（内存）
     - 哈希算法：SHA-256（替代默认的 SHA-1，提高安全性）
   - **风险**：索引管理增加系统复杂度
   - **缓解**：使用成熟的 LangChain Indexing API，提供降级方案（直接添加）

5. **状态管理**：Python TypedDict + JSON持久化
   - 简单直接，易于调试
   - 风险：大规模状态管理性能问题
   - 缓解：只持久化关键字段

6. **事件管理**：Python queue + threading
   - 标准库，无需额外依赖
   - 风险：多线程调试复杂
   - 缓解：单线程事件循环

#### 技术风险矩阵

| 风险项 | 影响 | 概率 | 严重性 | 缓解措施 |
|:---|:---|:---|:---|:---|
| LLM API故障 | 高 | 中 | 高 | 降级方案（反射路径），本地缓存 |
| 向量检索不准确 | 中 | 中 | 中 | Query Rewrite优化 |
| 状态持久化失败 | 中 | 低 | 中 | 定期备份，异常处理 |
| 内存泄漏 | 高 | 低 | 高 | 限制历史长度，定期清理 |

#### 性能指标

**响应时间目标**：
- 反射路径：< 100ms（本地处理）
- 简单对话：< 1.5s（LLM调用）
- 复杂推理：< 5s（包含RAG检索）

**资源消耗目标**：
- 内存：< 500MB（不含向量数据库）
- CPU：空闲时 < 5%，活跃时 < 30%
- 存储：向量数据库 < 100MB（MVP阶段）

## 8. RAM/ROM 记忆架构（V1.6 新增，V1.5 升级）

### 8.1 核心理念
为了解决 System Prompt 上下文过载和用户画像一致性问题，采用 RAM/ROM 分离架构：

- **RAM (State/Context)**: 
  - **结构化画像 (`user_profile`)**: 包含姓名、常住地、核心偏好等。**必须默认加载**，以满足 System Prompt 必须包含 Profile 的规则。存储为 `user_profile.json`。
    - **新闻偏好（v1.5 新增）**：`user_profile["news_preferences"]` 属于 RAM 结构化画像的一部分，包含：
      - `categories`: 新闻类别列表（如 ["finance", "technology", "sports"]）
      - `keywords`: 关键词列表（如 ["AI", "Artificial Intelligence", "大模型"]）
      - `excluded_topics`: 排除主题列表（如 ["娱乐八卦", "明星绯闻"]）
      - `update_frequency`: 更新频率（如 "daily"）
      - `last_updated`: 最后更新时间戳
    - **动态更新机制**：当用户口头表达新闻偏好时，`MemoryManager` 提取结构化偏好并更新 `user_profile.json`，确保实时生效。
  - **短期记忆 (`history`)**: 当前对话上下文。
  - **实时信息 (`context_signals`)**: 时间、传感器状态。

- **ROM (Tool)**:
  - **向量记忆 (`query_user_memory_tool`)**: 检索过往的非结构化片段、具体经历、次要偏好。存储在 ChromaDB。
  - **记忆写入 (`save_user_memory_tool`)**: 保存新的事实或偏好。

### 8.2 关键变更
1. **Profile 独立管理**: `user_profile` 从向量库中剥离，改为独立的 Pydantic 模型和 JSON 持久化，确保读写的一致性和确定性。
2. **Tool-based Memory**: 长期记忆检索不再是每次对话的默认动作，而是作为 Tool 由 LLM 按需调用（或由 Plan Node 规划），减少 Token 消耗并提高检索相关性。
3. **Tool Node 增强**: `tool_node` 在执行记忆类工具时，会自动注入最近的对话历史给 `MemoryManager` 进行 Query Rewrite，确保指代消解的准确性。
4. **新闻偏好管理（v1.5 新增）**：
   - 新闻偏好属于 RAM 结构化画像的一部分，存储在 `user_profile.json` 中，而非向量数据库。
   - 支持动态更新：当用户表达新闻偏好时，`MemoryManager` 提取并更新 `user_profile["news_preferences"]`。
   - 冲突检测：如果新偏好与现有偏好冲突（如用户说"以后我不想看体育了"），使用 `MemoryManager.detect_and_resolve_conflicts()` 处理。
   - 实时生效：新闻播报前，从 `user_profile` 中读取偏好，用于筛选和排序新闻内容。
