## 📍 核心设计原则

### **Base 地（常住地）vs 临时位置**

Animus 维护两种地点信息：

1. **Base 地（常住地）**：用户的长期居住地
   - 只有明确的搬家行为才会更新
   - 关键词：`["常住", "住在", "定居", "搬家", "搬到", "搬去", "搬迁", "安家", "落户"]`

2. **临时位置（当前位置）**：用户的短期停留地
   - 出差、旅游、临时访问
   - 关键词：`["出差", "旅游", "现在在", "正在", "临时", "来到", "去了"]`

---

## ✅ 当前逻辑（已优化）

### 1. 地点分类规则

```python
# 常住地关键词：明确表示长期居住
is_home = any(kw in new_fact for kw in ["常住", "住在", "定居", "搬家", "搬到", "搬去", "搬迁", "安家", "落户"])

# 临时位置关键词：明确表示短期停留
is_travel = any(kw in new_fact for kw in ["出差", "旅游", "现在在", "正在", "临时", "来到", "去了"])

# 如果两者都不明确，默认为临时位置（保守策略）
if not is_home and not is_travel:
    is_travel = True
```

### 2. 冲突解决规则

| 新信息类型 | 已有记忆 | 行为 |
|-----------|---------|------|
| 常住地 | 旧的常住地 | ✅ 覆盖（更新 Base 地） |
| 常住地 | 临时位置 | ✅ 保留两者（不冲突） |
| 临时位置 | 常住地 | ✅ 保留两者（不冲突） |
| 临时位置 | 旧的临时位置 | ✅ 覆盖（更新当前位置） |

## 🎯 设计哲学

### 保守策略

**原则**: 宁可多次确认，也不要误改 Base 地

**理由**:
1. Base 地是用户的核心画像信息，误改会导致严重的上下文错误
2. 临时位置可以频繁更新，不会造成太大影响
3. 如果用户真的搬家了，会明确说"搬家"、"搬到"等词

**示例**:
- ❌ 错误：用户说"我在北京"就改 Base 地 → 可能只是临时访问
- ✅ 正确：用户说"我搬家到北京了"才改 Base 地 → 明确的永久性迁移

## 🔄 与去重逻辑的配合

### 去重逻辑会处理

- ✅ "用户所在地是上海" vs "用户住在上海市" → 去重为一条
- ✅ "用户常住地是上海" vs "用户住在上海" → 去重为一条

### 冲突检测会处理

- ✅ "用户常住地是上海" + 新增"用户搬家到北京了" → 删除上海，保留北京
- ✅ "用户现在在北京出差" + 新增"用户现在在深圳出差" → 删除北京，保留深圳

### 两者不冲突

- ✅ "用户常住地是上海" + "用户现在在北京出差" → 两者共存
